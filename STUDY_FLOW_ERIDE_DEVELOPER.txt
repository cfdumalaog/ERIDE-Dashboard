E-RIDE DEVELOPER STUDY FLOW
Goal: Become productive enough to maintain and build this E-Ride project.

Suggested pace: 8 to 12 weeks if studying part-time, faster if full-time.
Main project stack to prepare for:
- Python + Streamlit for the business dashboard
- Excel formulas / financial modeling for backup costing sheets
- Git + GitHub for version control and deployment
- Kotlin + Android Studio for passenger and rider mobile apps
- React + TypeScript for future admin/operator dashboard
- Node.js / Express or NestJS for backend APIs
- PostgreSQL or similar database for trips, users, riders, fleet, and payments
- Google Maps Platform APIs for maps, places, geocoding, and routing
- AWS basics for backend deployment, monitoring, storage, and cost control

LOCAL PROJECT SETUP
Target: 1 day

1. Clone or open the repository.

2. Create a Python virtual environment:
   python -m venv .venv

3. Activate it on Windows PowerShell:
   .venv\Scripts\Activate.ps1

4. Install dependencies:
   pip install -r requirements.txt

5. Run the dashboard:
   python -m streamlit run eride_simulation_dashboard.py

6. Check the Excel fallback model:
   Open eride_operating_model.xlsx

7. Before editing, always check:
   git status -sb

8. After editing, validate:
   python -m py_compile eride_simulation_dashboard.py

9. If Streamlit changed:
   run the dashboard locally and click through the tabs.

AI TOOL SETUP AND SAFE USE
Target: 1 day

Recommended AI tools:
1. Codex / ChatGPT
   - Use for code explanation, debugging, refactoring, test writing, and documentation.

2. GitHub Copilot or Cursor
   - Use for autocomplete and small code generation.

3. Perplexity or web search
   - Use for current pricing, SDK docs, and API changes.

Rules for using AI safely:
1. Never paste secrets:
   - API keys
   - database passwords
   - production credentials
   - private tokens

2. Ask AI to explain before editing when you are learning.

3. Use AI for:
   - understanding code
   - generating first drafts
   - writing tests
   - finding bugs
   - summarizing docs
   - comparing implementation options

4. Do not blindly accept AI output.
   Always run the app, test the change, and inspect the diff.

5. Ask AI for small patches.
   Better:
   "Add a new charging station rent input to the Streamlit dashboard."
   Worse:
   "Rewrite the whole project."

6. Keep an AI work log:
   - prompt used
   - file changed
   - reason for change
   - test result

7. For pricing, laws, SDK docs, and cloud costs:
   ask AI to verify against official sources.

Useful AI prompts:
1. "Explain this function line by line like I am new to Python."
2. "Trace how this dashboard input affects monthly profit."
3. "Find possible bugs in this fare calculation."
4. "Create a Kotlin data class for an E-Ride trip."
5. "Write Retrofit API interfaces for these endpoints."
6. "Review this git diff and tell me what changed."
7. "Create unit tests for this fare calculation."
8. "Convert this business formula into Excel and Python."
9. "Explain this Android error and suggest a fix."
10. "Create a checklist before pushing this change."

PHASE 4 - FINANCIAL MODELING AND BUSINESS LOGIC
Target: 1 week

Learn:
1. Revenue
2. Fixed cost
3. Variable cost
4. Gross revenue per rider/day
5. Wage cost per rider/day
6. Break-even fare
7. Break-even rides/day
8. Cost per completed ride
9. Revenue per active user
10. Fleet utilization

Important E-Ride formulas:
- completed rides =
  min(users x rides/user/month, riders x rides/rider/day x operating days) x completion rate

- fare =
  base fare + distance charge + time charge - electric discount

- profit =
  revenue - rider payroll - incentives - energy - fleet - development - cloud/API - operations - fees/reserves

Mini task:
- Build three operating cases:
  1. Conservative
  2. Profitable
  3. Aggressive expansion

Checkpoint:
- You can explain why rider wage is not the only issue; utilization is the key driver.

FINAL TARGET

You do not need to know everything before helping.

Your first developer role in this project should be:
Dashboard and business-model developer first,
then backend/API contributor,
then Kotlin Android mobile/admin dashboard contributor,
then cloud/deployment support.

That path matches the current E-Ride project best because the operating model is already the main decision tool, and it teaches the business logic needed before building the production ride-hailing system.
