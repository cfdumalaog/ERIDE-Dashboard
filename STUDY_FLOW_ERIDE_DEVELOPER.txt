E-RIDE DEVELOPER STUDY FLOW
Goal: Become productive enough to maintain and build this E-Ride project.

Suggested pace: 8 to 12 weeks if studying part-time, faster if full-time.
Main project stack to prepare for:
- Python + Streamlit for the business dashboard
- Excel formulas / financial modeling for backup costing sheets
- Git + GitHub for version control and deployment
- React Native for passenger and rider mobile apps
- React + TypeScript for future admin/operator dashboard
- Node.js / Express or NestJS for backend APIs
- PostgreSQL or similar database for trips, users, riders, fleet, and payments
- Google Maps Platform APIs for maps, places, geocoding, and routing
- AWS basics for backend deployment, monitoring, storage, and cost control


PHASE 1 - PROJECT ORIENTATION
Target: 2 to 3 days

What to understand:
1. What E-Ride is trying to answer:
   - How many rides/day are needed to operate profitably?
   - How much revenue per user is needed?
   - How do fleet, rider wages, dev cost, cloud/API cost, and operations affect profit?

2. Current repo files:
   - eride_simulation_dashboard.py
     Main Streamlit dashboard.
   - eride_operating_model.xlsx
     Excel fallback model with formulas.
   - requirements.txt
     Python dependencies.
   - E-Ride_Formal_Proposal.md
     Business and technical proposal.

3. Current dashboard sections:
   - Client summary
   - Cost breakdown
   - Cloud tiers
   - Assumptions

Mini task:
- Run the dashboard locally:
  python -m streamlit run eride_simulation_dashboard.py

Checkpoint:
- You can explain why the model becomes profitable only when rider utilization is high enough.

PHASE 4 - FINANCIAL MODELING AND BUSINESS LOGIC
Target: 1 week

Learn:
1. Revenue
2. Fixed cost
3. Variable cost
4. Gross revenue per rider/day
5. Wage cost per rider/day
6. Break-even fare
7. Break-even rides/day
8. Cost per completed ride
9. Revenue per active user
10. Fleet utilization

Important E-Ride formulas:
- completed rides =
  min(users x rides/user/month, riders x rides/rider/day x operating days) x completion rate

- fare =
  base fare + distance charge + time charge - electric discount

- profit =
  revenue - rider payroll - incentives - energy - fleet - development - cloud/API - operations - fees/reserves

Mini task:
- Build three operating cases:
  1. Conservative
  2. Profitable
  3. Aggressive expansion

Checkpoint:
- You can explain why rider wage is not the only issue; utilization is the key driver.

PHASE 6 - GIT AND GITHUB WORKFLOW
Target: 3 days

Important:
- Do not commit Excel lock files like:
  ~$client_discovery_questionnaire.xlsx
  ~$eride_operating_model.xlsx

Common workflow:
1. Check changes:
   git status -sb

2. Review changed file:
   git diff -- eride_simulation_dashboard.py

3. Stage only intended files:
   git add eride_simulation_dashboard.py eride_operating_model.xlsx

4. Commit:
   git commit -m "Describe the change"

5. Push:
   git push origin main

Checkpoint:

FINAL TARGET

You do not need to know everything before helping.

Your first developer role in this project should be:
Dashboard and business-model developer first,
then backend/API contributor,
then mobile/admin dashboard contributor,
then cloud/deployment support.

That path matches the current E-Ride project best because the operating model is already the main decision tool, and it teaches the business logic needed before building the production ride-hailing system.
